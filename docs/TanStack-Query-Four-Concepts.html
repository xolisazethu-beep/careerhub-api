Server state vs client state

Four things useQuery gives you for free — and what the user suffers without them.

Client state is data your app owns: the search box text, which job is selected, the sort order. Server state is data that lives on a server and you only borrow: the job listings. It can be stale, shared across screens, slow to arrive, or fail. useQuery exists because server state needs four behaviours that useState + useEffect never give you.

What useQuery gives you freeWhat the user suffers if you hand-roll it
Loading & error as first-class values
isPending, isError, error		 You wire three state variables and reset them in the right order. Miss one path and the user stares at a blank screen — no spinner, no message — when the request hangs or fails.
Request deduplication
via the queryKey		 Ten components needing jobs fire ten identical requests. The browser hammers the API on every screen, wasting bandwidth and slowing every mount.
Caching
(stale-while-revalidate)		 A manual useEffect([]) re-fetches from scratch each mount, so every navigation flashes a spinner for data the user saw seconds ago.
Refetch on focus / reconnect + retries None of this happens by hand. Leave the tab for an hour → stale listings; one network blip → a permanent error with no self-recovery.

data renamed to jobs. It is undefined until the fetch resolves, then holds the array. You never set it by hand.
isPending
true while the first fetch is in flight and no cached data exists yet. Drives the skeleton.
isError
true if the queryFn threw. Drives the error panel.
error
The actual Error object that was thrown — its .message is shown to the user.
refetch
A function that re-runs the same query. Wired to the “Try again” button.
queryKey: ["jobs"]
The cache identity (see Section 2). Any component using this key shares this one fetch.
queryFn: fetchJobs
The async function that does the actual HTTP call (see Section 3). TanStack Query calls it for you.

Those five returned values are the “free” behaviours made visible. The render below picks exactly one of three branches from them — the user can only ever be in one state.

The queryKey contract

The cache identity: how results are stored, looked up, deduplicated and invalidated.

The queryKey is an array that acts as a primary key for the cache. The rule is simple:

Same key ⇒ same data  ·  changed key ⇒ a new query and a new fetchEvery component that calls useQuery with ["jobs"] reads from the one cached entry. The first one fetches; the rest get the cached array instantly. That is request deduplication and caching, both keyed off this array.
src/app/page.tsx — the key in context38 } = useQuery({39 queryKey: ["jobs"], // ← cache identity for the listings40 queryFn: fetchJobs,41 });

Two ways to get the key wrong

✗ Accidentally sharing a keyA jobs list and a single job detail both keyed ["jobs"] overwrite each other — whichever request resolves last wins.
Symptom: the list page shows one job (or crashes on .map), or the detail page renders a whole array of garbage.
Right vs wrong — sharing// ✗ WRONG — collision: both write the same cache slotuseQuery({ queryKey: ["jobs"], queryFn: fetchJobs });useQuery({ queryKey: ["jobs"], queryFn: () => fetchJob(id) });// ✓ RIGHT — distinct, hierarchical keysuseQuery({ queryKey: ["jobs"], queryFn: fetchJobs });useQuery({ queryKey: ["jobs", id], queryFn: () => fetchJob(id) });
✗ Using a unique key when it should be shared["jobs", Math.random()] never matches a previous key, so every mount is treated as a brand-new query.
Symptom: a skeleton flashes on every navigation and the cache never warms up — the exact problem TanStack Query was meant to solve.
The unstable-key trap// ✗ WRONG — new key every render, cache can never hituseQuery({ queryKey: ["jobs", Math.random()], queryFn: fetchJobs });// ✓ RIGHT — stable key, cache hits across mountsuseQuery({ queryKey: ["jobs"], queryFn: fetchJobs });

The mental model

1 · Look up
On mount, TanStack Query hashes the key and checks the cache for an entry.
2 · Hit or fetch
Cached & fresh → return it instantly. Missing/stale → run queryFn once.
3 · Dedupe
Other components with the same key in flight? They wait on the one request.
Rule of thumbPut everything the data depends on into the key — and nothing that changes randomly. ["jobs"] for the list, ["jobs", id] for a detail, ["jobs", { type, sort }] if the server filters. Stable, descriptive, and as granular as the data.

Why fetch does not throw on HTTP errors

A 404 or 500 is a successful round-trip — so you must throw yourself.

fetch only rejects on a network-level failure: DNS lookup fails, the device is offline, CORS blocks it, or the request is aborted. A 404 or 500 still resolves — the server was reached and answered, which from fetch's point of view is success. The status lives on res.ok (true only for 200–299).

The trapRemove the res.ok check and TanStack Query sees a resolved queryFn, so it stores the error body as success data. The UI leaves loading and renders the success branch against junk — an empty grid, blank fields, or jobs.map is not a function crashing the page. The error state never shows.

The code (src/lib/api.ts)

src/lib/api.ts — the network layer10 export async function fetchJobs(): Promise<JobListing[]> {16 const baseUrl = process.env.NEXT_PUBLIC_API_URL ?? "";17 const res = await fetch(`${baseUrl}/api/jobs`);1825 if (!res.ok) {26 throw new Error(`Failed to load jobs — server responded with ${res.status} ${res.statusText}`);27 }2829 return (await res.json()) as JobListing[];30 }
: Promise<JobListing[]>
The return type is the contract: this resolves to an array of jobs, or it throws. TanStack Query relies on exactly that shape.
baseUrl … ?? ""
Honour NEXT_PUBLIC_API_URL when set (a real backend); otherwise fall back to "" so the relative /api/jobs route is used — instead of fetching undefined/api/jobs, which 404s.
await fetch(...)
Fires the request. This line only rejects on a network-level failure — never on a 4xx/5xx.
if (!res.ok)
The critical line. res.ok is false for any non-2xx status. This is the only place an HTTP error becomes a thrown error.
throw new Error(... ${res.status} ...)
Throwing rejects the queryFn promise → TanStack Query flips isError and stores this Error. Including the status makes the message diagnostic, not generic.
return (await res.json())
Reached only on a 2xx. Parse the body and hand the typed array back as data.
With the checkHTTP errors are routed into the isError / error path → the user sees the red panel with a real message and a retry button (the Section 1 error UI).
Without the checkThe error body (e.g. {"error":"DB down"}) is treated as data. jobs.map(...) runs on a non-array → page crash, and the carefully-built error UI never appears.

Stale-while-revalidate

Refresh in the background; keep the old data on screen until the new arrives.

The default staleTime is 0, meaning data is considered stale the moment it arrives. When you return to the tab, TanStack Query triggers a background refetch — but the data already on screen stays rendered. The user sees the existing grid, not a spinner; only the background isFetching flag flips, and new data swaps in seamlessly when it lands.

src/app/providers.tsx — one client for the whole app21 export default function Providers({ children }) {22 const [queryClient] = useState(() => new QueryClient());2324 return (25 <QueryClientProvider client={queryClient}>26 {children}27 <ReactQueryDevtools initialIsOpen={false} />28 </QueryClientProvider>29 );30 }
useState(() => new QueryClient())
Creates the cache once per browser session. The initialiser never re-runs on re-render, and it isn't module-level — so one user's cache can't leak into another's server response.
QueryClientProvider
Makes that single client (and its cache) available to every useQuery below it. This is what lets ["jobs"] be shared app-wide.
ReactQueryDevtools
Dev-only overlay to watch queries flip between fresh, stale, and fetching — the SWR cycle made visible.

SWR vs the naive approaches

ApproachWhat the user sees on return to the tab
useQuery (SWR, default)Existing grid stays on screen. A background refetch runs (isFetching flips); fresh data swaps in with no flash.
useEffect(…, [])Runs once on mount and never again — never refreshes on focus. Stale data forever until a full reload.
Naive manual fix
(clear data, show spinner)		Flashes a loading state over content the user is reading. Jarring, and it throws away good data to show a spinner.

The SWR cycle, step by step

1 · Cached render
Return to tab → the previously fetched grid renders immediately from cache.
2 · Revalidate
Because staleTime: 0, a background refetch starts. isFetching → true. UI unchanged.
3 · Swap
Fresh data resolves and replaces the old seamlessly. isFetching → false. No spinner shown.
