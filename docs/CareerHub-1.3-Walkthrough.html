🧭 The one-sentence summary of Assignment 1.3

We took the list of jobs out of the JavaScript file it was hardcoded in, put it behind a web address (/api/jobs), and taught the page to ask for it over the network using a library (TanStack Query) that handles loading, errors, retrying and caching for us — without changing a single one of the visual components from the earlier assignments.

Part 1

Written Decisions

Four conceptual questions answered before writing any code. These explain why TanStack Query, not just how.

1️⃣ Server state vs client state — four things useQuery gives you for free

What you get freeIf you hand-rolled it with useEffect + useState, and what the user would suffer
Loading & error as first-class values
isPending, isError, error		You'd wire three state variables and reset them in the right order. Miss one path and the user stares at a blank screen with no spinner and no message when the request hangs or fails.
Request deduplication
via the queryKey		Ten components needing jobs fire ten identical requests. The browser hammers the API on every screen, wasting bandwidth and slowing every mount.
Caching (stale-while-revalidate)A manual useEffect([]) re-fetches from scratch each mount, so every navigation flashes a spinner for data the user saw seconds ago.
Refetch on focus / reconnect + retriesNone of this happens by hand. Leave the tab for an hour → stale listings; one network blip → a permanent error with no self-recovery.

2️⃣ The queryKey contract

The queryKey is the cache identity — how the cache stores, looks up, deduplicates and invalidates a result. Same key ⇒ same data; changed key ⇒ a new query and a new fetch.

  • ✗Accidentally sharing a key: a jobs list and a single job detail both keyed ["jobs"] overwrite each other — whichever resolves last wins. Symptom: the list page shows one job (or crashes on .map), or the detail page renders a whole array of garbage.
  • ✗Using a unique key when it should be shared: ["jobs", Math.random()] never matches, so every mount is treated as new. Symptom: a skeleton flashes on every navigation and the cache never warms up — the exact problem TanStack Query was meant to solve.

3️⃣ Why fetch does not throw on HTTP errors

fetch only rejects on a network failure (DNS, no connection, CORS, abort). A 404 or 500 is a successful round-trip — the server was reached and answered — so the promise resolves and res.ok is false (true only for 200–299).

Remove the res.ok check and TanStack Query sees a resolved queryFn, so it stores the error body as success data. The UI leaves loading and renders the success branch against junk — an empty grid, blank fields, or jobs.map is not a function crashing the page. The error state never shows. Throwing on !res.ok is what routes HTTP errors into the isError path.

4️⃣ Stale-while-revalidate

Default staleTime is 0, so returning to the tab triggers a background refetch — but the data already on screen stays rendered. The user sees the existing grid, not a spinner; only the background isFetching flag flips, and new data swaps in seamlessly.

A useEffect(…, []) fetch runs once on mount and never again, so it never refreshes on focus — stale data forever until a full reload. The naive manual fix (clear data, show a spinner) would flash a loading state over content the user is reading. SWR gives freshness without the flash.

Explain it to a beginner Think of TanStack Query as a smart assistant for fetching data. Instead of you personally writing sticky-notes for “am I still loading?”, “did it fail?”, “did I already fetch this?”, the assistant tracks all of that. The queryKey is just the label on the drawer where it files each result — same label means “you already have this, here it is instantly”; a different label means “go fetch a fresh one.”

Part 2

Next.js Route Handler — the mock backend

A tiny server endpoint that answers GET /api/jobs with the seed listings as JSON. This is the seam a real production API will slot into later.

src/app/api/jobs/route.ts
1import { NextResponse } from "next/server";
2import { JOBS } from "@/lib/seed-jobs";
3import type { JobListing } from "@/types";
4
5/** GET /api/jobs — returns the seed listings as JSON. GET only. */
6export async function GET() {
7  const jobs: JobListing[] = JOBS;
8  return NextResponse.json(jobs);
9}
L1NextResponse is Next.js's helper for building HTTP responses. NextResponse.json() serialises an object and sets the Content-Type: application/json header for us.
L2We import the JOBS array from seed-jobs.ts — the same seed data from Assignment 1.1. The component tree no longer imports this directly; only the server does.
L3type-only import of JobListing. Typing the variable means if the seed data ever drifts from the contract, the build fails here — not in the user's browser.
L6The key line. Exporting a function literally named GET is how the App Router maps this file to the GET method of the /api/jobs URL. Because no POST/PUT is exported, any other method automatically gets a 405 Method Not Allowed.
L7Annotating jobs: JobListing[] asserts the seed matches the contract the frontend expects.
L8Return the JSON response. This is the entire backend for now — a real database call would replace this one line without touching anything else.
Why this matters Before this file, the jobs lived inside a React file and could only change by editing code and redeploying. Now they live behind a web address. The front-end fetches them like it would from any real server, which means the day a real API exists, we change one URL — not the UI.
Effect on the UI Directly, none — this is a server endpoint with no visual output of its own. Indirectly, it is the data source the entire job grid now renders from. You can prove it by opening http://localhost:3000/api/jobs in the browser and seeing the raw JSON array — every field of JobListing present, all four employment types represented, and the Luno DevOps listing with "isActive": false.

  • ✓Returns the Assignment 1.1 seed listings via NextResponse.json()
  • ✓Responds to GET only (no other method exported ⇒ 405)

Explain it to a beginner Imagine a vending machine. Before, the snacks were glued inside the app's code. Now we built a little hatch with a label that says “GET /api/jobs”. When anyone knocks with a GET request, the hatch hands back the list of jobs as JSON (a universal text format for data). Knock with any other kind of request and the hatch stays shut (error 405). That hatch is a “route handler.”

Part 3

Install & Configure TanStack Query

Create the provider that gives every component access to the query cache, and wire it into the root layout — keeping the client boundary in exactly one file.

# installed once
npm install @tanstack/react-query @tanstack/react-query-devtools
src/app/providers.tsx
1"use client";
2
3import { useState, type ReactNode } from "react";
4import { QueryClient, QueryClientProvider } from "@tanstack/react-query";
5import { ReactQueryDevtools } from "@tanstack/react-query-devtools";
6
7export default function Providers({ children }: { children: ReactNode }) {
8  const [queryClient] = useState(() => new QueryClient());
9
10  return (
11    <QueryClientProvider client={queryClient}>
12      {children}
13      <ReactQueryDevtools initialIsOpen={false} />
14    </QueryClientProvider>
15  );
16}
L1"use client" marks this as a Client Component. TanStack Query uses React state and the browser, so its provider must run on the client. Keeping this directive here means the layout can stay a Server Component.
L7The component accepts and forwards a children: ReactNode prop — it wraps whatever the app renders.
L8The most important line. The QueryClient is created with the useState initialiser form — useState(() => new QueryClient()). The function runs once per mount and never again on re-render, so each browser session gets exactly one client.
L11QueryClientProvider puts that client into React context so any useQuery deep in the tree can reach the same cache.
L13ReactQueryDevtools with initialIsOpen={false} mounts the floating debug panel, collapsed by default.
Why not a module-level const queryClient = new QueryClient()? A module-level client is created once on the server and shared across every incoming request. One user's cached jobs (or, in a real app, their private data) could leak into another user's response. The useState initialiser scopes the client to a single browser session and creates it exactly once — correct on both server and client.
src/app/layout.tsx — the wiring (Server Component, no "use client")

Part 4

Environment Variables & API Layer

A configurable base URL kept out of version control, and one pure function that performs the HTTP call — with no React anywhere near it.

.env.local (git-ignored)
NEXT_PUBLIC_API_URL=http://localhost:3000

.env* is listed in .gitignore, so this file never gets committed. The NEXT_PUBLIC_ prefix is what makes the value readable in browser code.

src/lib/api.ts
1import type { JobListing } from "@/types";
2
3export async function fetchJobs(): Promise<JobListing[]> {
4  const baseUrl = process.env.NEXT_PUBLIC_API_URL ?? "";
5  const res = await fetch(`${baseUrl}/api/jobs`);
6
7  if (!res.ok) {
8    throw new Error(
9      `Failed to load jobs — server responded with ${res.status} ${res.statusText}`
10    );
11  }
12
13  return (await res.json()) as JobListing[];
14}
L1Only a type import — no React, no components. This keeps fetchJobs pure network logic that can be unit-tested or reused anywhere.
L4Read the base URL from the environment. The ?? "" fallback means that if the env var is missing, we use a relative /api/jobs — the app still works against its own backend instead of fetching undefined/api/jobs and 404-ing.
L5Build the full URL and await fetch. This is the actual network request.
L7–11The crucial check. fetch does not throw on 404/500, so we test res.ok ourselves and throw a descriptive Error that includes the status code. This is what pushes HTTP failures into TanStack Query's isError path (see Part 1, Q3).
L13Parse the JSON and assert it as JobListing[] — the typed value useQuery will hand to the components as jobs.
Why a separate file with no React Keeping the network call out of the component means it has one job and can be tested in isolation. The component asks “give me jobs”; how that happens (which URL, error handling) is this file's concern alone. Swap the data source later and the components never notice.
Effect on the UI Nothing visible yet — this file isn't wired to a component until Part 6. But it is the function that powers all three UI states: its successful return becomes the job grid, its thrown Error becomes the red error panel's message, and the time it spends awaiting is what the skeleton fills. It compiles cleanly under npx tsc --noEmit.

✅ Requirements met

  • ✓.env.local with NEXT_PUBLIC_API_URL, git-ignored via .env* in .gitignore
  • ✓src/lib/api.ts exports a single function fetchJobs
  • ✓Reads the base URL from process.env.NEXT_PUBLIC_API_URL and builds the full URL
  • ✓Calls fetch, checks res.ok, throws a descriptive Error including the HTTP status code
  • ✓Returns parsed JSON typed as JobListing[]; imports nothing from React
Explain it to a beginner fetchJobs is a courier. You give it nothing; it goes to the address written on the env-var sticky note, knocks (fetch), and comes back with the jobs. If the address answers with “Not Found” or “Server Error,” the courier doesn't quietly hand you an empty box — it raises its hand and says exactly what went wrong (the thrown Error with the status code), so the app can show a proper “try again” screen instead of pretending it worked.
