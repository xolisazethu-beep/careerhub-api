The big picture

CareerHub has three layers. The browser talks to the UI; the UI talks to the API; the API talks to the database. The API and the database run inside Docker; the UI runs with Node.

🟢 In plain words

Think of a restaurant: the browser is the customer, the UI is the waiter, the API is the kitchen, and the database is the pantry. The customer never walks into the pantry — they ask the waiter, who tells the kitchen, who fetches from the pantry. Each layer only talks to its neighbour.

The two connections we had to wire

  1. API → Database: a connection string tells the API where Postgres is and how to log in.
  2. UI → API: an environment variable tells the UI the API's address, plus a CORS rule on the API so the browser is allowed to call it.

Step 1 — The database (PostgreSQL in Docker)

Postgres runs as a Docker service. It's defined in the API project's docker-compose.yml:

services:
  db:
    image: postgres:17
    container_name: careerhub24-pg
    environment:
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: password123
      POSTGRES_DB: CareerHub24
    ports:
      - "5544:5432"          # host 5544  ->  container 5432
    volumes:
      - careerhub24_pgdata:/var/lib/postgresql/data   # data survives restarts
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres -d CareerHub24"]

ℹ Why a volume?

The named volume careerhub24_pgdata stores the database files on disk, so your jobs, accounts and applications are not lost when the container restarts. This is what makes it a real database, not a throwaway.

Step 2 — The API (ASP.NET) reaches the database

The API is the only thing that talks to Postgres. It's told where the DB is via a connection string, injected in docker-compose.yml:

  api:
    build: { context: ., dockerfile: Dockerfile }
    container_name: careerhub24-api
    depends_on:
      db: { condition: service_healthy }   # wait for Postgres to be ready
    environment:
      ASPNETCORE_URLS: http://+:8080
      # Host is the SERVICE NAME "db" and the CONTAINER port 5432 (not 5544) —
      # containers talk to each other over the compose network by service name.
      ConnectionStrings__DefaultConnection: "Host=db;Port=5432;Database=CareerHub24;Username=postgres;Password=password123"
      Jwt__Key: "dev-only-careerhub-signing-key-...-32+chars"
    ports:
      - "8080:8080"

ℹ The key detail beginners miss

Inside Docker, the API reaches Postgres at Host=db;Port=5432 — the service name and the internal port. The 5544 mapping is only for you (tools on your laptop). Two containers on the same compose network find each other by name.

How the API uses that connection string (EF Core)

// Program.cs — register the DbContext with the connection string
builder.Services.AddDbContext<CareerHubDbContext>(o =>
    o.UseNpgsql(builder.Configuration.GetConnectionString("DefaultConnection")));

// ...on startup, create/upgrade the schema and seed sample data:
using (var scope = app.Services.CreateScope())
{
    var db = scope.ServiceProvider.GetRequiredService<CareerHubDbContext>();
    await db.Database.MigrateAsync();   // applies EF migrations to the real DB
    await SeedData.SeedAsync(db);       // inserts SA companies + jobs if empty
}

Step 3 — Point the UI at the API

The UI needs the API's address. We put it in .env.local at the root of this Next.js project:

# .env.local  (loaded by Next.js at build & dev time)
NEXT_PUBLIC_API_BASE_URL=http://localhost:8080

ℹ Why the NEXT_PUBLIC_ prefix?

Only variables that start with NEXT_PUBLIC_ are sent to the browser. The job board reads on the server, but auth/apply run in the browser — so the API URL must be public. Next.js inlines this value into the JavaScript at build time.

Reading jobs — on the SERVER (no browser request)

The /jobs page is a Server Component. It fetches from the API while rendering on the server, so the job data is already inside the HTML the browser receives:

// src/lib/jobs-api.ts
export const JOBS_API_BASE =
  process.env.NEXT_PUBLIC_API_BASE_URL ?? "http://localhost:8080";

// src/app/jobs/(board)/page.tsx  — async Server Component
const res = await fetchJobsApi(`/api/jobs?page=1&pageSize=12`); // cache:"no-store"
const payload = await res.json();
const jobs = payload.data;          // straight from PostgreSQL, via the API

Auth & apply — from the BROWSER

Signing in and applying happen in the browser, so they call the API directly with fetch (through a small retry wrapper that rides out the API's cold start):

// src/lib/applicant-api.ts
const res = await fetchWithRetry(`${API_BASE}/api/v1/auth/login`, {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ email, password }),
});
const { token } = await res.json();   // a real JWT, signed by the API

Step 4 — Let the browser call the API (CORS)

A browser will block a page on localhost:3000 from calling localhost:8080 unless the API explicitly allows that origin. That permission is CORS, configured on the API:

// Program.cs
builder.Services.AddCors(o => o.AddPolicy("CareerHubFrontend", p => p
    .WithOrigins("http://localhost:3000")   // allow the UI's origin
    .AllowAnyHeader()                        // allow Authorization, Content-Type…
    .AllowAnyMethod()                        // GET, POST, PATCH…
    .AllowCredentials()));

app.UseCors("CareerHubFrontend");            // runs BEFORE auth in the pipeline
6Verify CORS is set (preflight)
$ curl -i -X OPTIONS http://localhost:8080/api/v1/auth/login \
     -H "Origin: http://localhost:3000" \
     -H "Access-Control-Request-Method: POST"
HTTP/1.1 204 No Content
Access-Control-Allow-Origin: http://localhost:3000   ✔
Access-Control-Allow-Methods: POST

🟢 In plain words

CORS is the API saying "I trust requests coming from localhost:3000." Without that line, the browser refuses to even send the request and you'd see "Failed to fetch".

🟢 The whole thing in one breath

Docker runs Postgres and the API; the API finds Postgres by the service name db via a connection string and creates the tables with EF migrations. The UI knows the API's address from NEXT_PUBLIC_API_BASE_URL, reads jobs on the server, and signs-in/applies from the browser — which the API allows through CORS. Every screen reads and writes the same real PostgreSQL tables through the API. That is the connection, end to end.
