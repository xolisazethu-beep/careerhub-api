CHAPTER 0

src — the whole front-end

One container, six rooms. Each has a single clear job.

src/
 ├─ app/ — pages & routes
 ├─ components/ — reusable UI blocks
 ├─ context/ — shared memory (login, toasts)
 ├─ fonts/ — the typefaces
 ├─ lib/ — logic, data, formatting
 └─ types/ — the data contract
A website is like a house. src is the whole house; each folder is a room with one responsibility. Good organisation is what lets a team build something large without chaos.

CHAPTER 2 · components

Reusable building blocks

REAL — Navbar.tsx & JobCard.tsx

  • A component is a LEGO brick — write once, reuse everywhere
  • Props are its inputs (which job, is it selected?)

CHAPTER 3 · context

Shared memory for the whole app

REAL — a live "Saved to your list." toast; navbar shows "Sign out"

SUMMARY · PART 1

The front-end folders

  • Separation of concerns — looks, logic, data shape & shared state each kept apart
  • Reuse — write a card once, draw it many times
  • Swap-readiness — demo data → real server with tiny changes
Click a card → page reads it (app) → drawn by bricks (components) → formatted by lib → shape guaranteed by types → login from context → in the right fonts. Every room plays its part.
11
SUMMARY · PART 2

Connecting to the C# back-end

  • HTTP carries the request & response
  • JSON is the shared language both sides read
  • The contract (types ↔ JobListingResponse.cs) keeps them in step
  • One seam — fetchJobs() — is the only door to the server
  • Back-end handles CORS + auth tokens for a secure, live connection

The UI was finished with mock data before the back-end existed — then flipped to live with a one-line change. That's the pay-off of clean separation.
