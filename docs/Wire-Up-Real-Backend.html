3 · Why direct calls (not a proxy)

The browser calls the backend straight from TanStack Query's queryFn, because the backend already returns CORS headers for the frontend origin (verified: an OPTIONS preflight returns 204 with Access-Control-Allow-Origin: http://localhost:3000). The env var is NEXT_PUBLIC_-prefixed, i.e. intended for the browser. No proxy route was needed, so nothing forwards or hides the URL — simplest correct choice for local dev.

4 · Manual steps YOU need to do / know

A. Prove the data is real (the acceptance test)

  1. Open the API docs UI: http://localhost:8080/scalar/v1
  2. Find POST /api/Jobs, fill in a new job (title, company, etc.) and send it.
  3. Refresh http://localhost:3000 — your new job appears in the grid. That round trip through Postgres is the proof it is not mock data.

B. Starting everything from cold (after a reboot)

# 1. Backend (API + Postgres) — from the backend repo folder:
cd C:\Users\Xolisa\assignment-3.2
docker compose up -d

# 2. Frontend — from this repo:
cd C:\Users\Xolisa\careerhub-1.3
npm run dev

The backend image takes a few minutes to build the first time (.NET 10 publish). After that it starts in seconds.

C. If the board shows the error panel / 500s: the Postgres container can drop into "recovery mode" when the machine is under heavy load (it happened once during the .NET build). The API's connection pool then holds dead connections. Fix: 
# wait for Postgres to report healthy, then bounce the API:
docker restart careerhub24-api
Give it ~30–60 s to re-run migrations and start listening, then refresh the page.

D. Known limitation — pagination

The backend holds ~600 jobs and paginates 20 at a time. The board currently requests ?page=1&pageSize=50, so it shows the first 50 and the search/sort/filter operate on those 50. Wiring real page navigation (using the links.next the API already returns) is a clean follow-up but was out of scope for "show live jobs".

E. Application & recruiter flows still use the in-app store

Submitting an application and posting jobs from the recruiter screens still write to this Next.js app's own /api routes, not the real backend — the live API exposes those under a different shape (e.g. POST /api/v1/jobs/{id}/applications). This is deliberate: only the job board was in scope. Wiring those to the real backend is a separate task.
