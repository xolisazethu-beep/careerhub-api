Connecting the CareerHub UI to the real C# API

Swapping the Next.js mock route for the live ASP.NET Core + PostgreSQL backend.

Generated 22 June 2026 · Frontend branch CAREERHUB-UI-1.3 · API branch assignment-3.2

What this achieves. The UI was reading 7 hard-coded jobs from an in-app mock route (/api/jobs). After these steps it reads 491 real listings from PostgreSQL via the C# API at http://localhost:5080, with no changes to any React component — only the network layer and a type definition changed.

7Completing the integration — detail endpoint & the apply flow done

Connecting the list endpoint exposed two gaps the lean list projection left behind. Both are now closed.

7a · The Learnership enum value (compile fix)

The API's JobType enum has a value the UI union lacked: Learnership. Adding it to EmploymentType broke three exhaustive Record<EmploymentType, …> maps (by design — they fail the build until the new key is supplied). All were updated:

  • src/lib/employmentType.ts — label "Learnership"
  • src/components/JobStatusBadge.tsx — a rose badge colour
  • src/lib/careerhub-store.ts — store label
  • src/components/FilterBar.tsx — a "Learnership" filter chip

7b · Detail endpoint — fetchJobById

Added JobListingDetailResponse (the API's detail DTO: it is the list shape plus description, minimumRequirements, companyProvince, companyWebsite) and a fetchJobById(id) in src/lib/api.ts that calls GET /api/jobs/{id} and hydrates the long-form fields the list could not supply.

GET http://localhost:5080/api/jobs/{id}
// → { ..., description: "…", minimumRequirements: "Matric.", companyProvince, companyWebsite }

7c · Live description in the SummaryPanel

When a job is selected on the home page, SummaryPanel now fetches its detail (same ["job", id] cache the apply page uses) and shows the real description and minimum requirements. Sections the API genuinely has no data for (responsibilities, a structured skills list, experience years) render only when populated — no empty headers.

7d · The apply flow now works with real jobs bug fixed

The bug: /apply/[jobId] previously resolved jobs only from the mock careerhub-store (old seed ids). With real API GUIDs on the board, every "Apply" click would have shown "Job not found."

The page now loads the job from the API via TanStack Query, with a graceful fallback to the local store for recruiter-posted demo jobs:

const { data: apiJob, isPending, isError } = useQuery({
  queryKey: ["job", jobId],          // shares SummaryPanel's cache
  queryFn: () => fetchJobById(jobId),
  retry: false,                      // a 404 fails fast → local fallback
});
const localJob = getJobById(jobId);  // recruiter-posted demo jobs
// → normalised into one ApplyJob shape; spinner while loading, "not found" only when both miss

Verified

  • npx tsc --noEmit → exit 0 (all exhaustive maps satisfied).
  • GET /api/jobs/{id} returns description + minimumRequirements.
  • /apply/{realGuid} renders 200 (was the "not found" path before).

8Full schema alignment — backend now matches the frontend done

Four fields the UI model expected had no home anywhere in the API. Rather than leave them as permanent empty defaults, they were added to the C# side so the contract is honest end-to-end.

FieldBackend change
responsibilitiesNew List<string> → PostgreSQL text[] column
skillsNew List<string> → PostgreSQL text[] column
minimumExperienceYearsNew int column (+ >= 0 check constraint)
applicantCountDerived — server-side COUNT over the Applications navigation (not stored)

Migration & reseed

dotnet ef migrations add AddJobListingStructuredFields --project CareerHub.Api.csproj
# Wipe the dev volume so the updated seed regenerates listings WITH the new fields:
docker compose down -v && docker compose up -d db
dotnet run --project CareerHub.Api.csproj   # auto-migrates + reseeds on startup
Gotcha: run with a build, not --no-build, right after creating a migration. --no-build runs the previously-compiled assembly, which lacks the just-generated migration + model snapshot, and EF Core 10 then throws PendingModelChangesWarning at startup.
