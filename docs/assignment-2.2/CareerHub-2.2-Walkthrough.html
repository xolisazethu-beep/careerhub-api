The big picture (read this first)

Architecture decision — why everything runs on the in-app API

For 2.2 the job board, detail, dashboard, apply, stats and close all read from this app's own API (src/app/api/*), backed by the persistent file store .data/careerhub-db.json plus the seed board. Two reasons:

  1. The close-loop is only demonstrable in one server. revalidateTag("jobs") clears Next.js's own server-side Data Cache. For closing a job on /dashboard to visibly invalidate /jobs, both pages must read from the store the PATCH mutates. The external Docker backend can't have a "close" endpoint added from this repo.
  2. It fixes the apply crash. The job-detail/apply pages used to depend on the external Docker API being awake; on a cold start the detail page would throw and you couldn't apply. Same-origin, always-on routes remove that dependency.

NEXT_PUBLIC_API_URL=http://localhost:3000 gives Server Components the absolute URL they need to fetch their own routes.

Part 2 — New route handlers

2a. Stats endpoint — api/applications/stats/route.ts

What was required

GET only (POST → 405). Return { jobId, applicationCount }[], one entry per job, with realistic non-zero counts. Empty array (not 404) when there are no jobs. Import the same mock jobs the [id] route uses — no duplication.

How I did it

I import the same JOBS seed array and the same listJobs/ countApplicationsByJob helpers the detail route uses, then map every job to a stat. The count is a realistic baseline plus live applications submitted via POST /api/applications, so the number climbs as candidates apply.

export async function GET() {
  const recruiterJobs = await listJobs();
  const recruiterStats = await Promise.all(recruiterJobs.map(async (j) => ({
    jobId: j.id,
    applicationCount: RECRUITER_BASELINE + (await countApplicationsByJob(j.id)),
  })));
  const seedStats = await Promise.all(SEED.map(async (j) => ({
    jobId: j.id,
    applicationCount: j.applicantCount + (await countApplicationsByJob(j.id)),
  })));
  return NextResponse.json([...recruiterStats, ...seedStats]);  // [] when no jobs — never 404
}
export async function POST() {
  return NextResponse.json({ title:"Method Not Allowed", status:405 },
    { status:405, headers:{ Allow:"GET" } });   // explicit 405
}

Important lines: the single NextResponse.json([...]) always returns an array; the explicit POST returns 405 with an Allow: GET header.

2b. Job status update — PATCH on api/jobs/[id]/route.ts

What was required

Add PATCH beside GET. Read id from params and status from the body. 404 (Problem Details) if no such job; 400 if status missing; otherwise update and return the job with 200. The change must persist for the server's lifetime.

How I did it

The "module-level mutable array" role is played by the persistent store, keyed by job id. A jobStatus map in server-store.ts records overrides; both GETs resolve the effective status as override ?? baseStatus.

export async function PATCH(request, { params }) {
  const { id } = await params;
  const recruiterJob = await getJob(id);
  const seed = recruiterJob ? null : SEED.find((j) => j.id === id);
  if (!recruiterJob && !seed) return problem("Job not found", ..., 404);

  let body = {};
  try { body = await request.json(); } catch {}
  if (typeof body.status !== "string" || !body.status.trim())
    return problem("Missing required field", ..., 400);

  await setJobStatus(id, body.status.trim());   // persists across requests
  return NextResponse.json(/* updated job detail, status echoed */);
}

UI effect

Nothing visible yet — but this is the hinge the whole assignment turns on: GET now reports a closed job as "Closed", which is what the candidate board and detail page will pick up after revalidation in Part 6.

In plain words

We built a tiny "database write": tell the server "this job is now Closed", it remembers it, and every read of that job afterwards says Closed.

Proving it changed

PATCH /api/jobs/<valid-id>  body {"status":"Closed"}  → 200 + updated job
GET   /api/jobs/<valid-id>                          → status now "Closed"
PATCH /api/jobs/<valid-id>  (no body)             → 400
PATCH /api/jobs/does-not-exist                    → 404
POST  /api/applications/stats                     → 405

Part 3 — Cache strategies with tags

What was required

Change the three jobs fetches from cache:"no-store" to a tagged cache, keep application data on no-store, and have both the candidate board and the dashboard share the "jobs" tag.

How I did it

Each jobs read now opts into the Data Cache and labels its response. Because Next 15 leaves fetch uncached by default, I pair the tag with cache:"force-cache" — the tag only governs data that is actually cached.

// jobs/(board)/page.tsx  —  candidate board
const res = await fetch(`${JOBS_API_BASE}${query}`, {
  cache: "force-cache",
  next: { tags: ["jobs"] },
});

// jobs/[id]/page.tsx  —  single detail (+ per-job tag, Stretch B)
const res = await fetch(`${JOBS_API_BASE}/api/jobs/${id}`, {
  cache: "force-cache",
  next: { tags: ["jobs", `job-${id}`] },
});

Application stats stay cache:"no-store" — applications arrive at any time with no clean employer trigger to invalidate on, so always-fresh is the only correct choice.

UI effect (production only)

On npm run build && npm start, the first visit to /jobs hits the API once; refreshes 2 and 3 are served from the Data Cache with no outbound fetch. Same for /jobs/[id]. npm run dev bypasses this cache on purpose.

Why two files, one tag, is correct

A tag isn't owned by a file — it's a label on a cached response. The candidate board and the dashboard table both label their jobs responses "jobs", so one revalidateTag("jobs") clears both. That's the intended behaviour: closing a job must update both surfaces.

Parts 4 & 5 — Parallel fetching + streaming

Part 4 — fetch jobs and stats in parallel

What was required

Add getApplicationStats(), fetch it alongside jobs with Promise.all (not on separate await lines), and join each row to its count in a new Applications column (0 when absent).
// ListingsTable.tsx
const [jobs, stats] = await Promise.all([getJobs(), getApplicationStats()]);
const countFor = (id) => stats.find((s) => s.jobId === id)?.applicationCount ?? 0;

Both requests leave the server together (within a few ms of each other in the Network waterfall) rather than one-after-the-other.

Part 5 — two independent Suspense boundaries

What was required

The page renders the heading immediately and hands each data source to its own <Suspense> with its own skeleton; the page itself performs no awaits and passes no data props.

How I did it

// dashboard/listings/page.tsx  — no awaits; renders shell, then streams
return (
  <section>
    <header>…All Listings…</header>
    <Suspense fallback={<ApplicationsSummarySkeleton />}><ApplicationsSummary /></Suspense>
    <Suspense fallback={<ListingsTableSkeleton />}><ListingsTable /></Suspense>
  </section>
);

Each component is an async Server Component that fetches its own data and ships an inline skeleton. ApplicationsSummary is the fast one (one fetch, no join); ListingsTable is slower (two fetches + a join), so it resolves second.

With a single boundary wrapping both, at T≈120ms you'd still see only skeletons — a boundary resolves only when everything inside it is ready, so the fast card would be held hostage by the slow table for ~330ms.

Part 6 — Server Action + useActionState

What was required

A "use server" action closeJobListing(prevState, formData) with a discriminated-union state, that PATCHes the job closed, calls revalidateTag("jobs") on success, and returns the closed job's title. A "use client" CloseJobButton using useActionState, wired into a new Action column.

The Server Action — runs only on the server

"use server";
import { revalidateTag } from "next/cache";

export async function closeJobListing(_prevState, formData) {
  const jobId = formData.get("jobId");
  if (!jobId) return { status:"error", message:"Missing job reference…" }; // no network call
  const res = await fetch(`${API_BASE}/api/jobs/${jobId}`, {
    method:"PATCH", headers:{ "Content-Type":"application/json" },
    body: JSON.stringify({ status:"Closed" }),
  });
  if (!res.ok) { const b = await res.json().catch(()=>null);
    return { status:"error", message: b?.detail ?? `HTTP ${res.status}` }; }
  const updated = await res.json();
  revalidateTag("jobs");          // clears the candidate board AND the dashboard
  revalidateTag(`job-${jobId}`);  // Stretch B: also the detail page
  return { status:"success", jobTitle: updated.title };
}

The Client Component — drives the form

"use client";
import { useActionState } from "react";
export default function CloseJobButton({ jobId, currentStatus }) {
  const [state, formAction, isPending] = useActionState(closeJobListing, null);
  if (currentStatus === "Closed") return null;
  if (state?.status === "success") return <span>Closed “{state.jobTitle}”</span>;
  return (
    <form action={formAction}>
      <input type="hidden" name="jobId" value={jobId} />
      <button disabled={isPending}>{isPending ? "Closing…" : "Close listing"}</button>
      {state?.status === "error" && <p>{state.message}</p>}
    </form>
  );
}

The PATCH is fired by the server inside the action — there is no client-side fetch to the PATCH endpoint in the browser Network tab. On the candidate's next /jobs load, the invalidated "jobs" tag forces a fresh fetch and the job shows Closed; its detail page swaps the apply form for the "no longer accepting applications" message.

In plain words

The employer clicks Close. The server quietly marks the job closed and rips up its cached copies of the jobs data. The button says "Closed ‹title›". Next time anyone opens the jobs page, it's rebuilt with the job showing Closed.

Bonus fix — applying for a job no longer crashes

Symptom: opening a job and trying to apply errored out / wouldn't let you through. Cause: the detail page and the apply form depended on the external Docker backend; on a cold start the page throws into its error boundary and the apply POST fails.

Fix: RealApplyPanel now gates on the in-app candidate session (useAuth → /api/auth) and submits to the in-app POST /api/applications (same store the stats endpoint counts). No external backend, no cold-start crash — and every submission bumps the dashboard's Total Applications.
