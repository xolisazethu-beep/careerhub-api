Part 1 — Written decisions

Full answers live in README.md. The headlines:

Q1 · Most damaging error state

The expired-session 401 during /apply/[jobId]. Five steps of effort are on the line; a blank screen there reads as data loss and causes abandonment. It is handled with a "Session Expired → Sign in" boundary — never a dead-end retry.

Q2 · Business rules — server vs client, and the duplicate

Rules 1–3 are enforced both sides; rules 4–5 are server-authoritative. Hiding the Apply button is not enough — the API also rejects a duplicate with 409, surfaced as a toast carrying the API's own message (a whole-submission conflict, not a single bad field).

Q3 · Highest type-drift risk

JobListingResponse — its numeric fields and the type enum. Confirmed by the generator (Part 3).

Part 2 — API error handling

Step 1 · The typed error foundation — DONE

src/lib/api-error.ts defines ApiError extends Error with status, code, fields, the getters isUnauthorized/isForbidden/isValidation, and parseApiError(res) which reads the RFC 7807 body (wrapped in try/catch — some errors have no body) and maps status → code.

const code =
  res.status === 422 || (res.status === 400 && hasFieldErrors)
    ? "VALIDATION"            // lift problem.errors → fields
    : statusToCode(res.status); // 401→UNAUTHORIZED, 403→FORBIDDEN,
                                // 404→NOT_FOUND, 409→CONFLICT, 5xx→SERVER_ERROR
Deviation, documented: the table maps only 422→VALIDATION, but ASP.NET model validation returns 400 with an errors map. So a 400 carrying field errors is also treated as VALIDATION — otherwise those messages would be lost.

Step 2 · API client uses parseApiError — DONE

Every authenticated function now does throw await parseApiError(res) instead of throw new Error(...). Public reads (fetchJobs, fetchJob) keep the generic Error — they surface at the route error boundary, not a form.

Proof: a duplicate application now shows the API's specific 409 message, not "Request failed: 409 Conflict".

Step 3 · Global error boundary — DONE

src/app/error.tsx — "use client", props {error, reset}, renders a heading + error.message + "Try again" (reset()) + "Go home". useEffect(() => console.error(error)) is the Sentry hook.

Step 4 · Route-specific boundaries — DONE

Four route boundaries, each declaring its persona and delegating to a shared RouteErrorScreen that branches on error.code:

FilePersona / 403 heading
src/app/apply/[jobId]/error.tsxJob Seeker Access Required
src/app/applications/error.tsxJob Seeker Access Required
src/app/recruiter/error.tsxEmployer Access Required (= /dashboard)
src/app/recruiter/jobs/[jobId]/error.tsxEmployer Access Required (applicants)
The rule, enforced: UNAUTHORIZED and FORBIDDEN never render a retry button — re-rendering reproduces the identical error. They offer Sign in / go back instead. NOT_FOUND links back to the list; only the fallback (validation/server/unknown) shows "Try again".

Step 5 · 422 field errors → the wizard — DONE

JobApplicationWizard's submit catch now detects err instanceof ApiError && err.isValidation, iterates err.fields, calls form.setError(), and jumps to the step owning the first invalid field via STEP_FIELDS — then a toast: "Please review your application — some fields need attention." A server field name is mapped to the wizard field the user edits (coverNote/coverLetter → motivationText).

Test 12 (src/test/ApplicationWizard.test.tsx) drives the whole wizard, mocks a 422 on submit, and asserts: it lands on the Job-specific questions step, shows the server message inline, and shows the review toast. Passing.
