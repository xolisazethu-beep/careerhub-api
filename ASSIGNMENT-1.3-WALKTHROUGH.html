3 · Part 2 — The mock backend (src/app/api/jobs/route.ts)

A Next.js Route Handler turns a folder into an HTTP endpoint. A file at app/api/jobs/route.ts that exports a function named GET becomes GET /api/jobs. I reuse the existing seed array so the data stays identical to Assignment 1.1, and return it with NextResponse.json().

export async function GET() {
  const jobs: JobListing[] = JOBS;     // same seed data from 1.1
  return NextResponse.json(jobs);      // → HTTP 200 + JSON body
}

Only GET is exported, so a POST/PUT automatically gets 405 Method Not Allowed. The seed covers all four employment types and one isActive:false listing, so every visual state from 1.2 is exercised.

4 · Part 3 — Installing & wiring TanStack Query

npm install @tanstack/react-query @tanstack/react-query-devtools

The QueryClient is the in-memory cache + scheduler. It must be available to every component that calls useQuery, so it's provided at the top of the tree. But the client must live per browser session, never as a shared module-level variable (that would leak one user's cache into another's response on the server). The fix: create it with the useState initialiser form.

providers.tsx — the client boundary

"use client";
const [queryClient] =
  useState(() => new QueryClient());
return (
 <QueryClientProvider client={queryClient}>
   {children}
   <ReactQueryDevtools
     initialIsOpen={false} />
 </QueryClientProvider>
);

layout.tsx — stays a Server Component

<body>
  <Providers>          {/* ← only this is new */}
    <AuthProvider>
      <ToastProvider>
        ...
      </ToastProvider>
    </AuthProvider>
  </Providers>
</body>

No "use client" here — the client boundary lives inside providers.tsx.

5 · Part 4 — Environment variable + the fetch layer

.env.local holds the API base URL so the URL isn't hardcoded in source. The NEXT_PUBLIC_ prefix is what makes it readable in the browser bundle.

NEXT_PUBLIC_API_URL=http://localhost:3000

src/lib/api.ts is pure network logic — no React, no components. The one subtle-but-critical line is the res.ok check:

const res = await fetch(`${baseUrl}/api/jobs`);
if (!res.ok) {
  // fetch does NOT throw on 404/500 — only on network failure.
  // Without this, an error body would be cached as "success".
  throw new Error(`Failed to load jobs — server responded with ${res.status} ${res.statusText}`);
}
return (await res.json()) as JobListing[];

7 · Part 6 — Replacing the hardcoded array (page.tsx)

The import of JOBS is gone. In its place, one hook:

const { data: jobs, isPending, isError, error, refetch } = useQuery({
  queryKey: ["jobs"],
  queryFn: fetchJobs,
});
ValueMeaningUsed for
jobsthe fetched array (undefined while loading)filtering, the grid, selected job lookup
isPendingfirst fetch in flight, no data yetshow <JobListSkeleton/>
isError / errorthe query failedshow error panel + error.message
refetchre-run the querythe "Try again" button

One more adjustment: the sessionStorage restore effect used to validate the stored id against the jobs array. On mount jobs is undefined, so that guard can never pass — I removed it. A stale id now simply leaves selectedJob as null and the summary panel stays absent. Graceful, no crash.
